域名的选择

  • 各种小众域名的坑
    • 🔒 .dev 域名被强制启用 HTTPS
    • .xyz、.test、.local
  • 公网测试环境请使用正规的 .com / .net 子域名。
  • 不要在测试环境中使用 .dev 域名，除非你已配置好有效的 TLS 证书并强制使用 HTTPS。
  • 避免使用冷门或免费 TLD（如 .xyz、.ga、.ml），它们可能被邮件系统、短信网关或防火墙拦截。
  • 本地开发用 .test 或 localhost；

在开发和测试阶段，很多开发者习惯使用 .dev 作为本地或测试环境的域名后缀，比如 myapp.dev 或 staging.myproject.dev。毕竟，“dev” 就是 “development” 的缩写，看起来再合适不过了。然而，这种做法在现代浏览器环境下可能会带来意想不到的问题——根本无法通过 HTTP 访问。

各种小众域名的坑

🔒 .dev 域名被强制启用 HTTPS

自 2019 年起，Google 将 .dev 顶级域名（TLD）加入到 HSTS 预加载列表（HSTS Preload List） 中。这意味着：

所有以 .dev 结尾的域名，浏览器会强制使用 HTTPS，即使你手动输入 http://xxx.dev，也会被自动重定向到 https://xxx.dev。

这是 Google 在 get.dev 官网上明确说明的：

“The .dev top-level domain is included on the HSTS preload list, making HTTPS required on all connections to .dev websites and pages without needing individual HSTS registration or configuration.”

换句话说，浏览器在发起请求前就已经决定：必须走 HTTPS，根本不会尝试 HTTP 连接。

假设你在 AWS Elastic Beanstalk 上部署了一个测试应用，并配置了 CNAME 记录指向 myapp.dev。当你在浏览器中访问 http://myapp.dev 时，页面可能直接超时或显示连接失败。

你可能会以为是 DNS 配置错误，但如果你在终端执行：

curl -I http://myapp.dev

却能正常返回 200 OK！

  • curl 不受 HSTS 影响：它不会读取浏览器的 HSTS 预加载列表，也不会自动跳转 HTTPS。
  • 浏览器会强制跳转：Chrome、Firefox、Safari 等现代浏览器看到 .dev 域名，会在本地直接内部重定向到 HTTPS（状态码 307），根本不会发出 HTTP 请求。

如果你的测试服务器没有配置 SSL 证书或未监听 443 端口，HTTPS 请求就会失败，导致“连接超时”或“SSL 错误”。

.xyz、.test、.local

你可能会想：那我换一个冷门后缀，比如 .xyz、.online 或 .fun？

但这些域名也有隐患：

  1. 邮件和短信 deliverability 问题

    • 某些邮件服务商（如 Gmail、Outlook）会将包含 .xyz 链接的邮件标记为垃圾邮件；
    • 有开发者反馈：仅在短信中包含 .xyz 域名，就会导致消息静默失败（用户收不到）。

  2. 防火墙或安全软件拦截

    • 很多企业防火墙会主动屏蔽 .xyz、.top 等“高风险”TLD；
    • 即使你的服务完全合规，也可能因域名后缀被误判为恶意网站。

  3. .test 和 .localhost 是保留域名

    • .test、.example、.localhost、.invalid 是 RFC 6761 官方保留用于测试的域名，不会被公共 DNS 解析，适合本地开发。
    • 但它们不适合公开测试环境（如 staging 服务器），因为无法通过公网 DNS 解析。
