前几天APP分发网址和APP分享网址突然被微信封禁，无法访问，申诉后微信给回了如下邮件内容

尊敬的用户： 很抱歉，你在申诉的网站api.example.com，经检测识别您申诉的网址仍然存在风险（关于网址规范详情请参考 https://urlsec.qq.com/eviltype.html），所以我们将对您申诉的网站我们继续提示风险。
本次审核不通过的原因为：网站存在恶意链接，包括但不限于违规链接如下：
https://api.example.com/app/release/134
https://api.example.com/app/share?uid=10243&type=app_share
请按照标准解决相关问题后再重新提交申诉，届时我们将重新审核您的申诉网站。

以下为总结的微信防封注意点：

域名被封解读

👉 这两个 URL 本身并不是“传统病毒/木马”那种恶意
👉 而是非常典型地触发了「接口型网址 + 可跳转 / 可传播 / 可下载」的风控规则
👉 在腾讯的规则里，这类 URL 默认就是高风险

你把「API 接口」直接暴露成了「可被用户点击、分享、访问的网页链接」，这是腾讯 URL 安全非常明确的雷区。
API 域名 = 永远不要出现在用户可点击场景

1️⃣ api.example.com/app/release/134

这个 URL 具有以下 高危特征组合：

特征 风控视角
api. 子域 接口域名
/app/release/134 像“版本 / 发布 / 安装包”
直接 HTTP GET 可访问 可被外部扫描 / 调用
无登录 / 无签名 任意人可访问

在腾讯的规则体系中，这类 URL 极容易被判定为：

  • 📌 疑似应用分发接口
  • 📌 疑似 APK / IPA / 更新入口
  • 📌 疑似灰产 App 投放通道

即使你真实返回的是 JSON，只要 URL 结构“像”，就会被风控命中。

2️⃣ api.example.com/room/share?...

这个 URL 命中了腾讯的多个高风险规则：

/share
?uid=1024
&type=app_share

在腾讯规则中，这种 URL = 🚨 重点打击对象

关键字 风险含义
share 分享传播
uid 用户维度传播
type=app_share App 拉新 / 裂变

👉 这就是标准的「裂变 / 跳转 / 诱导传播 URL 模板」

腾讯的 URL 安全非常明确地把以下行为列为高风险：

  • App 分享落地页
  • 带参数的跳转链接
  • 用户维度的传播 URL
  • 可在 QQ / 微信中被直接点击

不管你内容是不是合规，只要结构像，就直接拦。

人工审核

以上解读是针对机审，当“举报量”和“访问量”过大时就会有人工检测介入，就要根据业务规则调整了。

当有用户对链接内容进行投诉举报时，在一定时间内，当举报次数达到一定数量时，就有工作人员接入，开始人工检测。

当链接访问量过大，并通过系统自动检查出该域名已加入白名单时，域名就会被提交给工作人员进行检测。

微信屏蔽域名后，我们就会在微信端看到“已停止访问该网页”的警示图片。同时，系统或人工就会对域名进行拉黑处理，并记录域名及其路径，进入black-list.之后，即使为该域名重新建站，但只要一用，就会被微信封闭。

整改

原 api 域名彻底禁止外部直接访问

api.example.com

❌ 不作为任何可点击 URL
❌ 不允许浏览器直接访问

技术措施：

  • 只允许：App 内调用、服务端调用

  • 必须校验：token、sign、UA

  • 浏览器访问直接返回：403 或固定 JSON（无跳转、无链接）

不要用 api. 做分享链接

正确做法：

share.example.com
www.example.com
m.example.com

然后：

  • 页面 允许 做的事

1、纯内容展示
2、静态文案
3、图片 / 介绍
4、明确的「用户主动点击」按钮

  • 页面 禁止 做的事（必封）

1、页面打开即跳转
2、自动拉起 App
3、自动跳下载
4、通过 JS 判断 UA 强制跳转
5、iframe 内嵌下载页

👉 一定要“用户点击后才发生行为”

URL 结构必须“去风控化”

  • ❌ 高危写法，必须避免的关键词（直接命中规则）
/share
/app_share
/jump_from
/release
/jump
/invite
/download
/install
/openapp
  • ✅ 低风险写法
/page
/info
/detail
/view

参数也要改：

- ?uid=1024&type=app_share
+ ?id=10240
+ ?content_id=10240

📌 原则：

  • 不体现传播行为
  • 不体现 App
  • 不体现跳转
  • 不体现用户裂变

重新发起申诉

你重新申诉时，可以这样写（建议照抄）：

我司已完成整改：

  1. api 子域名已不再作为任何可点击访问的网址，仅供 App 内部接口调用
  2. 所有分享与落地页面已迁移至独立展示域名，不包含自动跳转、下载或诱导行为
  3. 已移除 app_share、jump 等高风险传播参数
    请复审，谢谢。

多域名落地页方案

为防止域名被封我们还可以采用落地页域名不固定的方案。

1、可随时更换分享域名

发起分享时从后台获取分享地址，可随时更换，不要放在前台拼接，后台直接生成。这样对于老版本的APP可随时更换分享域名。

2、分享域名和落地页域名分离

把A域名，作为分享域名； 把B域名，作为落地域名（也就是用户访问看到的域名）； 只有从A域名跳转到B域名，B域名显示的才是我们想展示给用户的内容，如果直接复制B的网址，打开的只是一个错误页面或者不违规的内容，也就是说如果TA点击右上角举报，举报的和访问的虽然是同一个网址，可是显示的内容却不同。从而最大限度的防止域名被封杀。

A 往 B 跳转时禁掉 refer 元素。

3、使用老站和知名站点的域名作为分享域名

其实是跳板原理。建站时间久和行业内相对知名的域名最不容易被微信封杀，当跳转代码注入到上述网站再利用安防网址做跳转，指向落地页，被微信封杀的可能性微乎其微。

4、建立落地页域名池

由分享页每次跳转不同的落地页，防止被封。

5、对接微信域名检测接口

微信域名检测接口，做到防患于未然。这个可以自己开发也可以购买。

检测到某个域名被封就将其从域名池中移除。
