第一部分：基础理论问答（附解析）

1. 脚本生命周期

题目：解释 Awake()、OnEnable()、Start()、Update()、LateUpdate()、FixedUpdate()、OnDisable()、OnDestroy() 的调用顺序与使用场景。

解析：
调用顺序大致为：

Awake()
OnEnable()
Start()
Update() / FixedUpdate()
LateUpdate()
OnDisable()
OnDestroy()
  • Awake()：对象被加载时调用（即使脚本未启用），适合做初始化数据。
  • OnEnable()：脚本启用时调用，可能多次触发（反复启用/禁用）。
  • Start()：脚本第一次启用并在第一帧更新前调用，通常初始化依赖其他对象的内容。
  • Update()：每帧调用，和帧率相关。
  • FixedUpdate()：物理更新帧调用，时间间隔固定（与 Time.fixedDeltaTime 一致）。
  • LateUpdate()：所有 Update() 之后调用，适合做相机跟随等需要确保前面逻辑执行完的任务。
  • OnDisable()：脚本被禁用时调用。
  • OnDestroy()：对象销毁前调用。

2. 协程与多线程区别

题目：协程和多线程的区别是什么？

解析：

  • 协程：单线程中分段执行任务，通过 yield 暂停与恢复，适合控制逻辑和时间流。
  • 多线程：真正的并发执行，需要线程安全处理，不能直接访问 Unity 主线程的对象。
  • Unity 协程不是并行，而是调度在主线程中的分段执行。

示例：

IEnumerator WaitExample()
{
    yield return new WaitForSeconds(1f);
    Debug.Log("After 1 second");
}

3. Prefab (预制体) 的优缺点

优点：

  • 复用性强，修改源 Prefab 会影响所有实例。
  • 动态加载，减少场景复杂度。

缺点：

  • 编辑器中修改实例可能导致不易维护的差异（Overrides）。
  • 如果资源依赖关系复杂，加载可能导致额外内存开销。

Prefab Variants：

  • 相当于继承的 Prefab，可以在保留原有结构的基础上做差异化。

4. 坐标系转换

题目：世界坐标、本地坐标、屏幕坐标、视口坐标区别？

解析：

  • 世界坐标 (World)：全局参考系下的位置。
  • 本地坐标 (Local)：相对于父对象的坐标。
  • 屏幕坐标 (Screen)：以像素为单位的 UI 屏幕位置（左下角为 (0,0)）。
  • 视口坐标 (Viewport)：归一化的屏幕坐标，范围 [0,1]。

示例：

Vector3 worldPos = Camera.main.ScreenToWorldPoint(Input.mousePosition);

5. OnTriggerEnter vs OnCollisionEnter

区别：

  • OnTriggerEnter：Collider 设置了 isTrigger，不产生物理碰撞，只触发事件。
  • OnCollisionEnter：实际发生碰撞，产生物理反应。

注意：两者都要求至少一方有 Rigidbody。

6. Resources vs Addressables

Resources：

  • 使用简单，但所有资源打包到一个文件，加载慢且占内存。
    Addressables：
  • 可异步加载，支持远程资源更新，更适合大项目。

7. Static Batching vs Dynamic Batching

  • Static Batching：适合静态物体，合并后减少 Draw Call。
  • Dynamic Batching：适合小型动态物体，有一定 CPU 开销。

第二部分：C# 与 Unity 特性（解析版）

1. 值类型 vs 引用类型

  • 值类型（struct, int, float）：存储在栈上，赋值时复制。
  • 引用类型（class, string, object）：存储在堆上，赋值时复制引用。

2. const / readonly / static

  • const：编译期常量，必须初始化。
  • readonly：运行时只可在构造函数内赋值。
  • static：类级别共享。
  • static readonly：运行时常量，全局共享。

3. 委托与事件

public delegate void MyAction();
public event MyAction OnSomething;

void Trigger()
{
    OnSomething?.Invoke();
}

4. struct （结构体）vs class

  • struct 不支持继承，值传递，适合小数据块。
  • class 引用传递，支持继承，适合复杂对象。
特性 struct（结构体） class（类）
类型类别 值类型（Value Type） 引用类型（Reference Type）
内存分配 通常分配在栈（stack）上（也可能被装箱到堆上） 分配在堆（heap）上
赋值行为 赋值时会复制整个结构体（深拷贝值） 赋值时只复制引用（浅拷贝）
继承 不支持继承其他 struct/class（但能实现接口） 支持继承和多态
默认构造函数 编译器提供隐式无参构造（不可自定义），字段会被默认初始化 可以定义无参或有参构造函数
空值（null） 不能为 null（除非是 Nullable<T>） 可以为 null
装箱（Boxing） 当转为 object 或接口类型时会发生装箱 不存在装箱问题
默认访问修饰符 默认是 private 默认是 private
析构函数 不支持析构函数 支持析构函数
可变性 通常建议保持不可变（immutable） 可变性更灵活

5. GC 触发

  • 频繁创建临时对象。
  • 使用 string 拼接产生大量新字符串。
  • LINQ 查询返回的枚举器。

问题1、Unity 中 GC 是如何触发的？有哪些典型的触发场景？

参考解析：

  1. 托管堆内存不足（分配新对象时，没有足够连续空间时触发）。
  2. 手动调用 System.GC.Collect()。
  3. 加载/卸载场景（会清理无引用的托管对象）。
  4. Profiler 连接（某些情况下 Unity 为了统计数据会额外触发 GC）。
  5. 频繁产生临时托管对象（如 string 拼接、foreach 迭代 List<T>、LINQ、装箱等）。
    • GC 是非实时的，可能在主线程中暂停游戏帧（Stop-the-world），增量 GC 会分帧处理减少卡顿。

问题2、如何减少 GC 触发次数？

参考解析：

  • 对象池（Object Pool）复用 GameObject 和托管对象。
  • 避免频繁产生临时对象（特别是在 Update() 里）。
  • 避免使用 string 拼接，改用 StringBuilder。
  • 避免频繁装箱/拆箱（boxing/unboxing），尤其是 struct 与 object/接口混用时。
  • 避免在热路径使用 foreach 遍历非数组集合（因为可能生成枚举器对象）。
  • 关闭或减少不必要的日志输出（Debug.Log 会产生 GC 压力）。
  • 合理设置引用，使无用对象及时失去引用，让 GC 有效工作。

第三部分：代码实操题（解析版）

1. 延迟执行

Invoke("DoSomething", 2f);

等价于：

StartCoroutine(Delay());
IEnumerator Delay()
{
    yield return new WaitForSeconds(2f);
    DoSomething();
}

2. 对象池优化

  • 避免频繁 Instantiate / Destroy。
  • 用 Queue 存储已创建的对象，反复利用。

3. 异步场景切换

IEnumerator LoadSceneAsync()
{
    AsyncOperation op = SceneManager.LoadSceneAsync("Game");
    op.allowSceneActivation = false;
    while (op.progress < 0.9f)
    {
        yield return null;
    }
    op.allowSceneActivation = true;
}

4. 四元数避免万向锁

transform.rotation = Quaternion.Euler(0, 90, 0);

欧拉角旋转会在特定角度失去一个自由度（万向锁），四元数可避免。

第四部分：性能优化与面试陷阱

1. 性能优化类

（1）Update 优化

你在一个场景里有 500 个敌人，每个敌人脚本都有 Update()，运行时帧率很低，你会怎么优化？

期望候选人答：

  • 集中管理：用一个 Manager 统一调度敌人逻辑。
  • 减少帧调用频率（按需更新 / 间隔更新）。
  • 关闭不必要的脚本（enabled = false）。

（2）GC 优化

如果游戏在运行过程中频繁出现 GC 卡顿，你会怎么排查和优化？

期望候选人答：

  • 用 Profiler 查看 GC 分配来源。
  • 避免频繁创建临时对象（例如 new / string 拼接）。
  • 用对象池复用。
  • 改用 StringBuilder、List.Clear() 而不是创建新对象。

（3）Draw Call 优化

你发现场景 Draw Call 数量很高，怎么降低？

期望候选人答：

  • Static Batching / Dynamic Batching。
  • 合并材质（Texture Atlas）。
  • 使用 GPU Instancing。
  • UI 上减少 Canvas 重绘次数。

（4）物理优化

物理计算很慢，有哪些优化方法？

期望候选人答：

  • 使用 Layer Collision Matrix 避免不必要的碰撞检测。
  • 减少碰撞体数量和复杂度。
  • 适当提高 Fixed Timestep 间隔。
  • 使用简单的 Collider（BoxCollider 代替 MeshCollider）。

2. 面试陷阱类

这些问题的关键是考查候选人是否知道 Unity 的一些“坑”，所以问的时候要有场景化。

（1）Destroy 延迟

如果你调用了 Destroy(gameObject)，它会立刻销毁吗？为什么有时我还能在本帧访问它？

期望候选人答：

  • 不会立刻销毁，实际销毁是在当前帧结束时执行。
  • 如果要立即销毁，可用 DestroyImmediate()（但很少推荐用）。

（2）协程引用丢失

如果一个对象正在执行协程，但对象被销毁了，协程会继续运行吗？

期望候选人答：

  • 不会继续运行，协程会被自动停止。
  • 如果想跨对象继续执行，需要挂载到其他对象（如全局管理器）。

（3）deltaTime 使用

为什么在 FixedUpdate 里用 Time.deltaTime 会有问题？

期望候选人答：

  • Time.deltaTime 是上一帧的渲染间隔，在 FixedUpdate 中应该用 Time.fixedDeltaTime。

（4）GetComponent 性能

如果一个脚本在 Update() 里频繁调用 GetComponent，会有什么性能问题？怎么优化？

期望候选人答：

  • GetComponent 会查找组件，有一定性能消耗。
  • 应该在 Awake() 或 Start() 缓存引用。
