BlurHash 占位图

每当前端从接口获取数据展示，当图片未下载完成前，会看到满屏的空白方框，并不美观，当然我们可以用懒加载图占位。

但有时同配色的模糊图占位(BlurHash 占位图方案)可能是更好的选择。

原理

BlurHash 能将一张图片转换成一个很短的字符串（仅 20–30 个字符！），用作该图片的占位符。你可以在服务端完成这一转换，并将这个字符串与图片一起存储。当你向客户端发送数据时，同时提供图片的 URL 和 BlurHash 字符串。客户端收到后，会将该字符串解码成一张模糊的占位图，在真实图片通过网络加载完成前先显示出来。这个字符串非常短，可以轻松嵌入你使用的任何数据格式中——例如，直接作为 JSON 对象中的一个字段。

该项目提供了各种语言的编码和解码方案供参考，使用很方便。

编码和解码速度如何？

这些实现并没有做太多优化。在处理非常大的图像时，可能会有点慢。对于相同输入或输出尺寸，编码器和解码器的性能大致相当，因此在 UI 线程上解码非常大的占位图也可能稍显缓慢。

不过！高效使用该算法的关键在于不要对全尺寸图像直接运行 BlurHash。因为图像的精细细节都会被丢弃，所以你应该先将图像缩小后再进行 BlurHash 处理。如果你正在生成缩略图，那就直接对缩略图运行 BlurHash，而不是原始大图。

同样地，在显示占位图时，使用非常小的图像（比如宽度仅为 32 像素甚至 20 像素）效果就非常好——只需让 UI 层将其放大即可，视觉效果与直接解码成完整尺寸几乎无法区分。

如何选择 X 和 Y 方向的分量数量？
这在一定程度上取决于个人偏好。分量越多，占位图保留的信息就越多，但 BlurHash 字符串也会越长。而且，并非分量越多看起来就越好。我们通常使用 4×3（X=4，Y=3），这个配置在效果和长度之间取得了不错的平衡。

不过，你应该根据图像的宽高比来调整分量数量。例如，非常宽的图像应使用更多的 X 分量、更少的 Y 分量。

Swift 示例项目中包含一个测试应用，你可以实时调整参数并查看效果。

某些实现中的 “punch” 参数是什么？
这是一个用于调整解码后图像对比度的参数。

  • 1 表示正常；
  • 小于 1 的值会让效果更柔和；
  • 大于 1 的值则会让模糊占位图的对比更强烈。

本质上，这是一个设计参数，让你能微调视觉风格。
从技术角度看，它通过缩放 AC 分量（交流分量）的大小来实现这一效果。

BlurHash 用途

  • 用作图片加载占位符
  • 无需昂贵模糊操作即可遮蔽图像：Mastodon 就用它来做内容警告遮罩。
  • 提取图像区域的平均颜色非常容易：数据表示方式使得你可以轻松估算图像顶部边缘、角落等区域的平均色。Swift 的 BlurHashKit 实现中就有相关实验代码。此外，整张图的平均颜色其实就是 DC 分量（直流分量），甚至无需实现复杂的 DCT 解码逻辑就能获取。
  • 尝试实现带色调的投影效果：我们曾设想利用 BlurHash 扩展 UI 元素的边界来生成有色投影，但目前还没时间真正实现。

如何传递解码尺寸

BlurHash 本身的设计中不直接在字符串里携带图像的原始尺寸（如 width / height），官方推荐做法：将尺寸作为额外字段与 BlurHash 字符串一起传输

BlurHash 的核心理念是极简、短小、通用。它的字符串只编码颜色和模糊结构信息，不包含元数据（如尺寸、格式等）。因此，最佳实践是在你的数据结构中单独传递尺寸信息。

例如，在 JSON 中：

{
  "image_url": "https://example.com/photo.jpg",
  "blurhash": "LKO2?U%2Tw=w]~RBVZRi};RPxuwH",
  "width": 1200,
  "height": 800
}

客户端拿到 blurhash 和 width/height 后，可以：

按比例显示占位图（避免布局偏移）

用宽高比设置容器尺寸，再将解码后的 BlurHash 图像拉伸填充
