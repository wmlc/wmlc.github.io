一、什么是 Unicode 双向控制字符？

Unicode 标准中定义了一组双向控制字符（Bidirectional Control Characters），用于控制文本在混合书写方向（如阿拉伯语从右到左 + 英文从左到右）环境中的显示顺序。

常见相关字符包括：

名称 缩写 Unicode 方向 作用
左至右标记 LRM U+200E LTR 插入一个左→右的方向性
右至左标记 RLM U+200F RTL 插入一个右→左的方向性
左至右嵌入 LRE U+202A LTR 强制后续文本左→右显示
右至左嵌入 RLE U+202B RTL 强制后续文本右→左显示
左至右覆盖 LRO U+202D LTR 强制后续字符按 LTR 排列（覆盖原方向）
右至左覆盖 RLO U+202E RTL 强制后续字符按 RTL 排列（覆盖原方向）
PDF 终止格式化 PDF U+202C - 结束 LRE/RLE/LRO/RLO 效果
左至右隔离 LRI U+2066 LTR 隔离并设置子段 LTR
右至左隔离 RLI U+2067 RTL 隔离并设置子段 RTL
首强隔离 FSI U+2068 auto 根据第一字符判断方向
PDI（结束隔离） PDI U+2069 - 结束隔离（LRI/RLI/FSI）

这些字符不可见，但会改变文本的视觉呈现顺序，从而制造“视觉欺骗”。复制后仍存在。

二、如何利用 RLO/LRO 发起 XSS 攻击？

攻击原理

攻击者利用 RLO（U+202E）将恶意代码“反转”显示，使其看起来像合法内容，从而绕过人工审核或简单关键词过滤。

示例：伪装成图片文件名

正常用户看到：

report.pdf

但实际字符串是：

"x\"\u{202E}fdp. gnirts\"=crs \"1(trela\"=rorreno "

但插入到 HTML 中却变成：

<img src="x" onerror="alert(1)" src="x‮fdp. gnirts"=crs "1(trela"=rorreno ">

由于 \u{202E} 的存在，浏览器将 fdp. 反向显示为 .pdf，造成“这是一个 PDF 文件”的错觉，而 onerror 属性却真实存在

示例：绕过 WAF / 黑名单 XSS 检测

攻击者可能发送这样的 payload（内部含 RLO）：

看起来：

"><script>alert(1)</script>

实际是：

"><\u202E tpircs><\u202E tpircs>alert(1)

浏览器渲染时恢复顺序 → 绕过 WAF / 黑名单 XSS 检测。

三、为什么传统过滤容易失效？

  1. 关键词过滤绕过：
    若仅过滤 onerror、<script>，但攻击载荷通过 RLO 被“隐藏”，过滤器可能看不到原始关键词。

  2. 视觉欺骗绕过人工审核：
    管理员看到的是 invoice.pdf，实际却是恶意脚本。

  3. 编码混淆：
    攻击者可用 \u202e、&#8238;、%E2%80%AE 等多种形式编码 RLO，增加检测难度。

四、PHP 防御策略

✅ 1. 输入阶段：移除或拒绝双向控制字符

在接收用户输入时，主动清理或拒绝包含双向控制字符的数据。

/**
 * 移除输入中的 Unicode 控制字符
 * User: wml
 * DateTime: 2025/12/8 10:41
 * @param $input
 * @return array|string|string[]|null
 */
public function removeControlCharacters($input): array|string|null
{
    if (is_string($input)) {
        // 匹配并移除常见的 Unicode 双向控制字符等
        return preg_replace('/[\x{200E}\x{200F}\x{202A}-\x{202E}\x{2066}-\x{2069}]/u', '', $input);
    }

    if (is_array($input)) {
        return array_map(function ($value) {
            return $this->removeControlCharacters($value);
        }, $input);
    }

    return $input;
}

// 使用示例
$filename = removeControlCharacters($_GET['filename'] ?? '');

⚠️ 注意：使用 u 修饰符确保 UTF-8 正确处理。

✅ 2. 输出阶段：始终进行上下文相关的 HTML 转义

即使输入已清理，输出时仍需转义：

use function htmlspecialchars;

$filename = removeControlCharacters($_GET['filename'] ?? '');
$safeFilename = htmlspecialchars($filename, ENT_QUOTES | ENT_HTML5, 'UTF-8');

echo "<img src='/uploads/{$safeFilename}' alt='{$safeFilename}'>";

这样即使有遗漏，" 也会被转为 &quot;，破坏 HTML 属性注入。

✅ 3. 内容安全策略（CSP）

通过 HTTP 头限制脚本执行：

header("Content-Security-Policy: default-src 'self'; script-src 'self'; object-src 'none';");

即使 XSS 注入成功，现代浏览器也会阻止 onerror 中的脚本执行（除非显式允许 'unsafe-inline'）。

✅ 4. 示例：针对文件上传场景

不要信任用户提供的文件名：

// 生成安全文件名
$originalName = $_FILES['file']['name'];
$extension = pathinfo($originalName, PATHINFO_EXTENSION);
$allowedExtensions = ['jpg', 'png', 'pdf'];

if (!in_array(strtolower($extension), $allowedExtensions)) {
    die('非法文件类型');
}

// 使用随机名存储
$safeFileName = bin2hex(random_bytes(16)) . '.' . $extension;
move_uploaded_file($_FILES['file']['tmp_name'], '/uploads/' . $safeFileName);

// 显示时也用安全名，不回显原始名
echo "文件已保存为: " . htmlspecialchars($safeFileName);

五、总结

防御层级 措施
输入 清理或拒绝双向控制字符
存储 不存储原始危险输入，使用安全标识符
输出 上下文相关的 HTML 转义（htmlspecialchars）
运行时 启用 CSP，限制脚本执行
设计 避免直接回显用户可控内容到 HTML 属性中

🔒 核心原则：永远不要信任用户输入，无论它看起来多“无害”。

Unicode 双向字符攻击属于“逻辑层欺骗”，它不依赖 <script> 标签，而是利用人类与机器对文本理解的差异。唯有通过纵深防御（Defense in Depth），才能有效抵御此类高级 XSS 变种。
