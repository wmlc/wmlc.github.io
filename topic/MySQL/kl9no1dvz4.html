数据库表名应该使用单数名词

  • 支持使用复数的理由
  • 支持使用单数的理由（更胜一筹）
    • 重点一：我们命名的是“关系”，而非“集合”
    • 重点二：在 SQL 的其他部分，单数读起来更自然
    • 重点三：与应用程序代码保持一致
    • 重点四：存在本身就是复数的实体，会破坏复数命名规则的一致性
  • 结论

在关系型数据库设计领域，一个经久不衰的争论是：数据库表的名字应该使用单数形式还是复数形式？ 例如，当你需要存储用户信息时，这张表应该命名为 user 还是 users？

这是一个看似简单，实则影响深远的设计决策。本文将探讨两种观点，并论证为何使用单数名词是更优的选择。

支持使用复数的理由

支持使用复数作为表名的观点通常基于以下两点：

  1. 语义直观：一张表里存储的是多个用户，因此用 users 这个复数名称更符合直觉。
  2. 在 FROM 子句中读起来顺畅：例如 SELECT id, name FROM users; 这句 SQL 读起来非常自然，就像在说“从用户们那里选择 ID 和姓名”。

这些理由听起来很有道理，但它们主要集中在查询的开头部分。

支持使用单数的理由（更胜一筹）

支持使用单数的理由更为深刻和根本，它们超越了表面的语法，触及了数据库的本质。

重点一：我们命名的是“关系”，而非“集合”

严格来说，我们不是在给一个“表”（table）命名，而是在给一个“关系”（relation）命名。数据库理论的核心是关系模型。

  • 我们定义的是“用户”这个实体的关系，即用户ID、姓名、地址等属性之间的关联。
  • 这个“关系”是唯一的，只有一个“用户关系”。
  • 我们只是利用这个单一的关系模式来存储多个用户的实例。

因此，用单数 user 来命名这个关系，更能体现其本质。

重点二：在 SQL 的其他部分，单数读起来更自然

虽然 FROM users 读起来不错，但一个查询远不止 FROM 子句。在 JOIN 和 WHERE 子句中，单数形式的优势就显现出来了。

-- 使用单数，连接条件清晰明了
SELECT id, name 
FROM user 
JOIN country ON user.country_id = country.id 
WHERE country.name = 'Canada';

-- 如果使用复数，连接条件就变得奇怪
-- SELECT id, name 
-- FROM users 
-- JOIN country ON users.country_id = country.id -- "users.country_id" 听起来像 users 这个整体有一个 country_id，而不是每个用户有一个

在 ON 子句中，user.country_id 表示“用户的国家ID”，逻辑清晰。而 users.country_id 则容易产生歧义，仿佛是“用户们”这个群体拥有一个共同的国家ID。

重点三：与应用程序代码保持一致

现代开发中，数据库通常与应用程序的代码（如Java、Python、Ruby等）通过ORM（对象关系映射）框架交互。

  • 在代码中，代表数据库记录的类（Class）几乎总是使用单数名词，例如 User、Order、Product。
  • 如果数据库表也使用单数 user，那么类 User 与表 user 之间的映射关系就非常直接和清晰。
  • 如果表使用复数 users，则会产生不匹配。更糟糕的是，许多ORM框架（如Ruby on Rails）会自动将类名复数化以匹配表名。这会导致一个尴尬的局面：如果你定义了一个类 Address，框架会试图去查找一个叫 addresss 的表（因为 Address 复数化规则不规则），这显然是一个错误。

重点四：存在本身就是复数的实体，会破坏复数命名规则的一致性

这是支持单数命名的最强有力的论据。

想象一下，你有一个名为 UserFacts 的类，用于存储用户的年龄、喜好颜色等零散信息。如果遵循“表名用复数”的规则，这张表应该叫什么？userfactss？这显然非常荒谬。

  • UserFacts 这个名字本身已经是一个复数概念。
  • 如果你坚持所有表名都必须是复数，那么对于 UserFacts 这样的实体，你将陷入困境，无法找到一个合适的复数形式，从而破坏整个数据库命名方案的一致性。

而如果采用单数命名规则，这个问题就迎刃而解了。你可以简单地将表命名为 user_fact 或 user_facts（这里 user_facts 被视为一个整体名词，而非复数），整个数据库的命名风格依然统一。

结论

虽然“表里存了多条数据，所以应该用复数”这个观点很直观，但从数据库理论、SQL查询的完整性、与应用程序的集成以及命名一致性来看，使用单数名词来命名数据库表是更合理、更优雅、更具可扩展性的选择。

它能确保：

  • 语义准确：正确反映“关系”的本质。
  • SQL清晰：在整个查询中，尤其是在连接和条件判断时，逻辑更通顺。
  • 代码一致：与应用层的类名完美对应，避免ORM的自动转换陷阱。
  • 方案健壮：能够优雅地处理那些名字本身就是复数的特殊情况，保持整个数据库命名体系的稳定和一致。

因此，下次为你的数据库表命名时，请优先考虑使用单数名词。
