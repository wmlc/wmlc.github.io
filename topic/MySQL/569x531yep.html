upsert

upsert 方法是 sql 语法 INSERT ... ON DUPLICATE KEY UPDATE 的封装, 表示先执行插入操作，若插入的纪录存在主键或唯一索引冲突，则会更新已存在的记录为指定的新值。

方法的第一个参数包含要插入或更新的值，而第二个参数列出关联表中唯一标识记录的列。方法的第三个也是最后一个参数是一个列数组，如果数据库中已存在匹配记录，则不插入改为更新这些列：

DB::table('users')->upsert(
    [
        ['email' => 'test@example.com', 'name' => '张三', 'age' => 18],
        ['email' => 'foo@example.com', 'name' => '李四', 'age' => 22],
    ],
    ['email'], // 唯一键（用来判断冲突）
    ['name', 'age'] // 冲突时需要更新的字段
);

等价于：

INSERT INTO `users` (`email`, `name`, `age`) VALUES 
('test@example.com', '张三', 18),
('foo@example.com', '李四', 22)
ON DUPLICATE KEY UPDATE `name` = VALUES(`name`), `age` = VALUES(`age`);

在上面的示例中，Laravel 将尝试插入两条记录。如果记录已存在相同的 email 列值，Laravel 将更新该记录的 name、age 列。

[! 警告] 除 SQL Server 外的所有数据库都要求 upsert 方法的第二个参数中的列具有「主键」或「唯一」索引。此外，MySQL 数据库驱动程序忽略 upsert 方法的第二个参数，并始终使用表的「主键」或「唯一」索引来检测现有记录。

📌 注意事项

  • 触发唯一键或主键冲突才会更新，普通非唯一字段重复不会触发。

  • 受影响行数：单条数据插入的场景

插入成功：返回 1

更新成功（值确实改变）：返回 2

更新未改变（值一样）：返回 0

自增跳号问题

在 MySQL 里，INSERT ... ON DUPLICATE KEY UPDATE 会导致 自增主键（AUTO_INCREMENT）即使发生了冲突也会加 1。

这是 MySQL 的设计机制导致的：

  1. 执行流程

    • 先尝试 INSERT 一条记录（这一步会消耗一个自增 ID）。
    • 如果违反唯一键/主键约束，则执行 UPDATE。
    • 已经分配的自增 ID 不会回收。

  2. 结果

    • 即使记录没插入成功（因为冲突改为更新），自增值依然加 1。
    • 所以会看到 ID 出现“跳号”。

📌 如何避免自增跳号？

严格来说 不能完全避免，但有几种替代方案：

  1. 不要依赖自增 ID 连续性（推荐做法）
    业务逻辑层面不要假设 ID 是连续的。ID 的本质是唯一标识，而不是序号。

  2. 先 SELECT 判断存在再决定 INSERT 或 UPDATE

    • 可以避免 ID 跳号，但会多一次查询，性能差。

  3. 使用 UUID / 雪花 ID / 自定义 ID

    • 如果业务非常在意 ID 连续性，可以放弃 AUTO_INCREMENT，改用分布式唯一 ID。

最佳实践

既要保证性能，保证在高并发下不要插入多条或出现锁冲突，又要解决自增ID跳号问题。

使用优先更新，如果没更新到再插入的思路：

实例：

if (!ChatMoney::query()->where(['type' => $type, 'date' => $date])->increment($field, $number)) {
    ChatMoney::query()->upsert(
        [
            ['type' => $type, 'date' => $date, $field => $number]
        ],
        ['type', 'date'],
        [$field => DB::raw("`$field` + $number")]
    );
}

这种写法还存在“极小概率”的高并发条件下触发唯一索引冲突，导致浪费一个自增ID，但可以忽略。

📌 这样写的优点

性能好：大多数情况下只会走一次 UPDATE，比“先查再插”的方案少了一次 SELECT。

语义清晰：“更新优先，不存在时插入”。

并发安全：即使高并发下两个请求同时走到 upsert，MySQL 唯一索引能保证不会插入重复行，后续会触发 update。

基本避免自增ID跳号： 极大降低了 ID 浪费的概率，因为大多数场景都能走 increment。
